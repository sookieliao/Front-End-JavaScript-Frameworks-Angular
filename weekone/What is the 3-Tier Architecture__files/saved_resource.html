          • OO Design is incompatible with Database Design

            • 2 comments
            • a year ago

            Tony Marston —

            I don't need to serialise an object's data, so I don't have the problem which you describe. I don't need to have a "Unique ID" field in any of my table classes as each class is tied to a single database table and instantiated into an object with a unique variable name. Each object can hold data from any number of database rows, and each row has its own unique primary key. If two tables are related to each other I don't bother with a relationship object. Each table object has two variables called $this->parent_relations and $this->child_relations which are both arrays containing any number of entries. If I need to know what relationships exist with any table I simply examine these variables. Again I don't need any Unique IDs as the table names are sufficient. All my data is passed around as arrays with the first level being indexed by row number and each row is an associative array of name=value pairs. My approach may not be as complicated as some developers seem to think is necessary, but as a follower of the KISS principle I start with "simple" and only add complications when absolutely necessary. My language of choice is PHP, and its simplicity is something which I have enjoyed for over a decade.

          • Not-so-SOLID OO Principles

            • 2 comments
            • a year ago

            Tony Marston —

            I am not confused about DIP simply because I have read the article written by the guy who invented it. His sample COPY program quite clearly shows that any benefits of using this principle only appear when you call an object which is replaceable at run-time, such as changing one device to another. In my applications if I have one domain object calling another domain object then I know what that object should be and I will never replace it with a different object. In this situation providing myself with the ability to do something that I will never actually do would be a complete waste of time.
            The only place where I use DI is when calling my Page Controllers. Each one of there can be used with any of my database table classes (I currently have over 400), but no class name is ever hard-coded in any Controller, it is injected at run-time.
            Object composition is only suggested as an alternative when you have deep inheritance hierarchies where one concrete class inherits from another concrete class, which I never do. My inheritance hierarchy is only one level deep - each concrete table class inherits from the same abstract table class. Thus I do not have the problem for which object composition is the solution, so why should I implement that solution?
